# 6. Proposed Methodology: BDPP-IoT

This section presents the proposed methodology, named BDPP-IoT, which extends the base paper's blockchain-based secure IoT data transaction framework. The base paper combines blockchain, IPFS, CP-ABE, CKKS homomorphic encryption, and game-theoretic pricing to support secure IoT data sharing. However, the base design still leaves three important limitations: centralized key authority, lack of practical key revocation, unchecked CKKS approximation error, and manually configured pricing parameters.

BDPP-IoT addresses these weaknesses by adding three main extensions:

1. Multi-Authority CP-ABE style access control with blockchain-based version revocation.
2. Operation-aware CKKS Error-Bound Reporting.
3. ML-calibrated dynamic pricing.

The methodology keeps the overall logic of the base paper intact. Data is still generated by IoT devices, encrypted before storage, referenced through IPFS CIDs, logged on blockchain, accessed through attribute-based policies, and priced before transaction completion. The difference is that BDPP-IoT improves the access-control logic, computation reliability, and pricing adaptability.

## 6.1 Methodological Motivation

The base paper depends on one Key Generation Center for attribute-based encryption. This means that if the KGC is compromised, the entire key infrastructure is exposed. It also means that revocation is difficult because users who already received keys may continue to hold usable keys. In a medical IoT environment, this is a serious issue. For example, a researcher who leaves a hospital should not continue to decrypt patient data.

The second issue is related to CKKS. CKKS is an approximate homomorphic encryption scheme. It supports encrypted computation over real-valued data, but the decrypted result is not guaranteed to be exact. The base paper uses CKKS for encrypted medical sensor computation, but it does not explicitly verify whether the approximate result stays within an acceptable error bound.

The third issue is pricing. The base paper's valuation model uses configurable constants and domain knowledge. This is useful for defining a theoretical model, but it does not adapt well to changing market behavior. In real data markets, price should be influenced by demand, integrity, freshness, seller reputation, and previous transaction behavior.

BDPP-IoT is designed around these gaps.

## 6.2 Proposed Components

### 6.2.1 Multi-Authority Access Control

Instead of relying on a single KGC, BDPP-IoT divides attributes across multiple independent authorities:

| Authority | Attribute Domain |
|---|---|
| AA_ORG | hospital_a, hospital_b, research_lab |
| AA_ROLE | doctor, cardiology_researcher, admin |
| AA_DATA | heart_rate, temperature, blood_pressure, glucose |
| AA_LOCATION | ward_a, ward_b, remote |

Each user key is represented as a collection of authority-specific attribute fragments. For example, a valid cardiology researcher may have:

```text
research_lab
cardiology_researcher
heart_rate
remote
```

The access policy used in the secure transaction demo is:

```text
research_lab AND cardiology_researcher AND heart_rate
```

Access is allowed only when the user's attributes satisfy the policy and the user's key version is still valid on the blockchain.

### 6.2.2 Blockchain Version-Based Revocation

BDPP-IoT binds each user to a blockchain version number. When the user is registered, the default version is:

```text
v_u = 1
```

When the user is revoked, the smart contract increments the version:

```text
v_u = v_u + 1
```

The decryption condition becomes:

```text
Access(u) = 1 iff Policy(S_u) = true AND v_key = v_chain
```

where `S_u` is the user's attribute set, `v_key` is the version carried by the user's issued key, and `v_chain` is the latest version stored on-chain.

This improves the base paper because access can be revoked without redesigning the entire access-control system.

### 6.2.3 Operation-Aware CKKS Error-Bound Reporting

BDPP-IoT uses CKKS through TenSEAL for encrypted computation. Because CKKS returns approximate results, BDPP-IoT verifies the decrypted result using an Error-Bound Reporting module.

For an input vector:

```text
M = {m_1, m_2, ..., m_n}
```

the expected value depends on the operation:

```text
Expected_mean = sum(M) / n
Expected_sum = sum(M)
Expected_variance = variance(M)
```

The percentage error is:

```text
Error% = |Observed_Result - Expected_Result| / |Expected_Result| * 100
```

The reliability decision is:

```text
Reliable = 1, if Error% <= threshold
Reliable = 0, otherwise
```

In the implementation, the normal TenSEAL CKKS result produced a very small error:

```text
normal_error_percent = 2.4136912231415975e-09
normal_reliable = 1
```

A high-noise/fault-injection scenario produced:

```text
high_noise_error_percent = 4.500000002522298
high_noise_unreliable_detected = 1
```

This shows that the EBR module can distinguish reliable encrypted-computation results from unreliable results.

### 6.2.4 ML-Calibrated Dynamic Pricing

The base paper uses static or manually configured pricing parameters. BDPP-IoT replaces this with an ML-based pricing layer trained on synthetic historical transaction records.

The feature vector is:

```text
x = (data_type, integrity_score, freshness, market_demand,
     seller_reputation, data_volume, access_frequency)
```

The target is:

```text
y = final_transaction_price
```

The implemented model is a Random Forest Regressor. It predicts a fair price:

```text
predicted_price = RF(x)
```

The pricing bounds for negotiation can be defined as:

```text
lower_bound = predicted_price * 0.85
upper_bound = predicted_price * 1.15
```

The implementation result shows that ML pricing significantly improves pricing accuracy:

```text
static_mae = 100.27005203097912
ml_mae = 11.217385659614619
static_r2 = -4.276097844000223
ml_r2 = 0.8966803787227509
```

This is an 88.8128 percent improvement over the static pricing baseline. The R2 result also shows that the static model fits the generated transaction-price behavior poorly, while the ML model explains most of the observed price variance.

## 6.3 End-to-End BDPP-IoT Pipeline

The proposed pipeline is:

1. Generate synthetic IoMT health records.
2. Compute metadata such as integrity score, freshness, demand, and reputation.
3. Create MA-CP-ABE policy and authority-based access fragments.
4. Encrypt the sensor vector using TenSEAL CKKS.
5. Store the encrypted payload in IPFS Kubo.
6. Store the IPFS CID on-chain using the Solidity smart contract.
7. Register valid and revoked users on-chain.
8. Revoke one user by incrementing their blockchain version.
9. Retrieve encrypted data from IPFS.
10. Perform CKKS encrypted computation.
11. Decrypt the approximate result.
12. Run EBR reliability checking.
13. Log reliability and transaction price on-chain.
14. Train and evaluate ML pricing.
15. Run ablation experiments to measure component impact.

The secure transaction demo confirms that the pipeline works end-to-end:

```text
ckks_backend = TENSEAL_CKKS
storage_backend = IPFS_KUBO
blockchain_backend = GANACHE_SOLIDITY
cid = onchain_cid
valid_user_allowed = 1
revoked_user_allowed = 0
onchain_reliable = 1
```
